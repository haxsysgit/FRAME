# The Goal Of FRAME Lab

The goal is simple:
find the smallest FRAME shape that stays useful across real software repos.

## What that means

The lab is trying to answer:
- what parts of FRAME survive real project pressure?
- what names are clear enough to keep?
- what fields actually help retrieval, checks, and agent behavior?
- what should be core versus tool-specific or optional?

## What winning looks like

A good FRAME line should be:
- useful to humans
- useful to tools
- portable across project families
- lean enough to maintain
- honest about proof and uncertainty

## What losing looks like

A bad FRAME line would:
- grow too many fields
- duplicate truth across files
- sound smart but not help decisions
- only fit one kind of repo
- make agents carry more paperwork than insight
